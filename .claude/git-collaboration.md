# Git 协作规范

## 分支策略

- 默认主分支为 `main`。
- 功能分支建议使用 `feat/<topic>` 或 `codex/<topic>`。
- 修复分支建议使用 `fix/<topic>`。
- 分支名使用小写短横线，例如 `feat/user-settings`、`fix/login-timeout`。
- 不在同一分支混合无关目标。

## 提交规范

提交信息建议使用一致格式：

```text
docs: 初始化 Agent 文档
feat: 添加用户设置页
fix: 修复登录超时处理
chore: 更新工具链配置
```

常用类型：

- `docs`：文档变更。
- `feat`：新增功能。
- `fix`：修复 bug。
- `refactor`：不改变行为的结构调整。
- `test`：测试相关。
- `chore`：工具链、配置、维护性变更。

## 提交前检查

提交前至少确认：

- `git status --short` 中只包含本次任务相关文件。
- 修改项目结构后更新了 `.claude/project-architecture.md`。
- 修复 bug 后更新了 `.claude/bug-fix-log.md`。
- 引入新技术或命令后更新了 `.claude/tech-stack.md`。
- 有可运行测试或验证命令时，已经执行并记录结果。

## Pull Request 规范

PR 描述建议包含：

- 背景：为什么需要这次变更。
- 变更：核心修改点。
- 验证：执行过的命令或人工检查。
- 风险：可能影响的流程、兼容性或后续工作。

## 代码评审重点

- 改动是否符合项目已有架构和命名风格。
- 是否引入了不必要依赖或过度抽象。
- 是否覆盖了关键边界条件和失败路径。
- 是否把个人环境、账号、密钥或临时路径写入仓库。
- 文档和实际目录结构是否一致。

## 禁止事项

- 不提交密钥、令牌、Cookie、账号凭据或私有数据。
- 不在未确认范围时执行破坏性 Git 命令。
- 不把无关格式化和功能变更混在一个提交里。
- 不依赖仅存在于个人机器上的绝对路径。
